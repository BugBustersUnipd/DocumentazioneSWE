\section{Architettura del Sistema (Layered Architecture)}

\subsection{Architettura Logica}
Il sistema è stato progettato seguendo un pattern architetturale di tipo Layered (a livelli). Questa separazione garantisce un alto grado di disaccoppiamento, facilitando la manuntenzione, i test e la scalabilità del sistema. L'architettura logica si divide in quattro livelli principali:

\subsubsection{Livello di Presentazione (Presentation Layer)}
Sviluppato in Angular (Single Page Application), è responsabile dell'interfaccia utente e dell'interazione con l'utente finale. Comunica con il backend tramite API RESTful.
\begin{itemize}
  \item Riceve gli input dell'utente (es. caricamento documenti, richieste di generazione testo)
  \item Comunica con il backend per inviare richieste e ricevere risposte.
  \item Gestisce il routing lato client e la renderizzazione reattiva dei dati.
\end{itemize}

\subsubsection{Livello applicativo e di Business Logic (Application Layer)}
Sviluppato in Ruby on Rails (API-only). Rappresenta il cuore del sistema e si struttura internamente in ulteriori sotto-livelli:
\begin{itemize}
  \item \textbf{Controller}: gestisce le richieste HTTP, orchestrando le operazioni necessarie per soddisfarle.
  \item \textbf{Service}: contiene la logica di business specifica, come l'integrazione con i modelli di intelligenza artificiale, la gestione dei documenti e la logica di elaborazione.
  \item \textbf{Model}: rappresenta le entità del dominio (es. Utente, Documento, Sessione) e gestisce la persistenza dei dati.
\end{itemize}

\subsubsection{Livello di Accesso ai Dati (Data Access Layer)}
Gestito dal modulo ActiveRecord di Ruby on Rails (ORM), si occupa della comunicazione con il database PostgreSQL. Fornisce un'interfaccia per eseguire operazioni CRUD (Create, Read, Update, Delete) sui dati persistenti traducendo le operazioni in query SQL sicure e ottimizzate, prevenendo vulnerabilità come le SQL Injection.

\subsubsection{Livello Dati (Data Layer)}
Costituito dal Database Relazionale (RDBMS) PostgreSQL. Ha la responsabilità di memorizzare in modo persistente tutte le informazioni necessarie al funzionamento del sistema, inclusi dati degli utenti, documenti caricati, sessioni di interazione e log delle attività.



\subsection{Architettura di Deployement (Monolite Containerizzato)}
Sebbene logicamente il sistema sia diviso in Frontend e Backend (decoupled), dal punto di vista del rilascio in produzione si è optato per un approccio Monolitico basato su Container.
Tutte le componenti del sistema sono pacchettizzate e distribuite insieme come un'unica entità logica su un singolo nodo ospite (Server/Macchina Virtuale), garantendo un ambiente isolato, riproducibile e coerente dalla fase di sviluppo fino alla produzione. L'orchestrazione locale è affidata a Docker Compose.

\begin{itemize}
  \item \textbf{Docker Host (Server Node)}: L'infrastruttura hardware o virtuale principale che ospita il motore Docker. Al suo interno risiede una Rete Virtuale Privata (Docker Network) che permette ai container di comunicare tra loro in totale sicurezza, senza esporre porte non necessarie all'esterno.
  \item \textbf{Container Frontend (Web Server)}: Un container dedicato all'esecuzione dell'applicativo Angular. Nelle richieste esterne, questo nodo funge da punto di ingresso per gli utenti, servendo i file statici della Single Page Application.
  \item \textbf{Container Backend (Application Server)}: Un container isolato basato su immagine Ruby che esegue il server web di Rails. È esposto internamente per ricevere le chiamate API dal frontend e processare la logica di business.
  \item \textbf{Container Database (Database Server)}: Un container basato sull'immagine ufficiale di PostgreSQL. Persiste i dati fisicamente scrivendoli su un Docker Volume esterno al container stesso, garantendo che i dati del sistema non vadano persi in caso di riavvio o aggiornamento del container.
\end{itemize}

\section{Requisiti Funzionali Soddisfatti}
Il periodo di analisi dei requisiti ha portato all'identificazione di un set di requisiti funzionali che il prodotto software deve soddisfare. 
Di seguito viene fornita una panoramica dei principali requisiti implementati nel sistema, con riferimento alle sezioni del documento dove sono descritti in dettaglio.
I requisiti funzionali si dividono in obbligatori, desiderabili e opzionali, a seconda della loro importanza e priorità per il progetto.
In particolare sono stati identificati 132 requisiti funzionali, di cui 97 obbligatori, 0 desiderabili e 35 opzionali.
I requisiti obbligatori hanno come focus l'implementazione dei tre moduli principali del sistema, ovvero il modulo di analisi documenti, il modulo di generazione testo e il modulo di raccolta dati, che rappresentano le funzionalità core del prodotto software.
Ciascuna di queste aree non era presente all'interno dell'applicativo Nexum quindi i rappresentanti dell'azienda (Eggon) hanno ritenuto fondamentale implementare queste funzionalità per soddisfare le esigenze degli utenti finali e garantire il successo del progetto.
I requisiti opzionali, invece, si concentrano su funzionalità come la gestione delle utenze e la distribuzione dei documenti, che invece sono già presenti all'interno dell'applicativo Nexum e quindi non rappresentano un elemento di differenziazione significativo per il prodotto software che stiamo sviluppando.
